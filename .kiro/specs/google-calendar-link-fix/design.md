# Google Calendar Link Fix - Bugfix Design

## Overview

The orchestrator LLM in the Strands Agents-as-Tools architecture paraphrases the `calendar_agent` tool result instead of passing the OAuth URL through verbatim. The trainer sees "click the link above" but no actual URL. The fix must structurally guarantee that OAuth URLs generated by `connect_calendar` reach the final WhatsApp response, bypassing LLM paraphrasing behavior. The approach is to detect OAuth URLs in the orchestrator's tool results during post-processing and ensure they are present in the final response text.

## Glossary

- **Bug_Condition (C)**: The orchestrator processes a calendar connection request, the `calendar_agent` tool returns a response containing an OAuth URL, but the orchestrator's final response text does not contain that URL.
- **Property (P)**: When the `calendar_agent` returns a response containing an OAuth URL, the final response delivered to the trainer MUST contain that same complete URL.
- **Preservation**: All non-calendar messages (student, session, payment, greetings) must continue to be routed and responded to exactly as before. Calendar error responses (no URL) must also be unchanged.
- **orchestrator**: The top-level `Agent` in `StrandsAgentService.process_message()` that routes queries to domain agents via the Agents-as-Tools pattern.
- **calendar_agent**: The `@tool`-decorated function in `_build_domain_agent_tools()` that calls `connect_calendar` and formats the OAuth URL response.
- **response_text**: The final string extracted from the orchestrator's `AgentResult` and returned to the trainer via WhatsApp.

## Bug Details

### Bug Condition

The bug manifests when a trainer requests calendar connection (e.g., "Conectar calendário google"). The `calendar_agent` tool correctly generates the OAuth URL and returns it as a string to the orchestrator. However, the orchestrator LLM generates a paraphrased response that references the link without including the actual URL. The system prompt instructions to "copy the response verbatim" are not reliably followed by the LLM.

**Formal Specification:**
```
FUNCTION isBugCondition(input, tool_results, final_response)
  INPUT: input of type str (trainer message),
         tool_results of type list (tool call results from orchestrator),
         final_response of type str (orchestrator output text)
  OUTPUT: boolean

  oauth_url := extractOAuthURL(tool_results)

  RETURN oauth_url IS NOT None
         AND oauth_url NOT IN final_response
END FUNCTION
```

### Examples

- Trainer sends "Conectar calendário google" → `calendar_agent` returns string containing `https://accounts.google.com/o/oauth2/v2/auth?client_id=...&state=abc123...` → Orchestrator responds "Clique no link acima para autorizar o acesso ao Google Calendar" → **BUG**: URL is missing from final response.
- Trainer sends "Conectar calendário outlook" → `calendar_agent` returns string containing `https://login.microsoftonline.com/common/oauth2/v2.0/authorize?client_id=...` → Orchestrator responds "O link de autorização foi gerado, clique nele para conectar" → **BUG**: URL is missing from final response.
- Trainer sends "Conectar calendário google" → `connect_calendar` fails (missing credentials) → `calendar_agent` returns error string "Google Calendar integration is not configured" → Orchestrator responds with error message → **NOT A BUG**: No URL was generated, error handling is correct.
- Trainer sends "Agendar sessão com João amanhã às 10h" → Routed to `session_agent` → **NOT A BUG**: No calendar connection involved.

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- Non-calendar messages (student management, session scheduling, payments) must continue to route to the correct domain agent and return correct responses.
- Calendar connection error responses (invalid provider, missing credentials, trainer not found) must continue to return appropriate error messages without URLs.
- General greetings and help requests must continue to be answered directly by the orchestrator without calling domain agents.
- OAuth state token storage in DynamoDB (trainer_id, provider, TTL) must remain unchanged.
- The `connect_calendar` tool function itself must remain unchanged.
- The 30-second timeout protection must remain unchanged.
- Conversation history injection must remain unchanged.

**Scope:**
All inputs that do NOT result in a `calendar_agent` tool call returning an OAuth URL should be completely unaffected by this fix. This includes:
- Student management messages
- Session scheduling messages
- Payment messages
- General greetings and help requests
- Calendar connection requests that fail (no URL generated)

## Hypothesized Root Cause

Based on the code analysis, the root cause is architectural — the system relies on the LLM to faithfully reproduce URLs from tool results, which is inherently unreliable:

1. **LLM Paraphrasing Behavior**: The orchestrator LLM receives the `calendar_agent` tool result (a string containing the OAuth URL) and generates a natural language response. LLMs tend to summarize/paraphrase tool results rather than copy them verbatim, especially for long URLs with query parameters. The system prompt instruction to "COPIE A RESPOSTA INTEIRA do calendar_agent" is a soft constraint that the LLM may ignore.

2. **No Structural URL Preservation**: The `process_message` method extracts `response_text` from the `AgentResult` object but performs no post-processing to verify that URLs from tool results are preserved. There is no fallback mechanism if the LLM drops the URL.

3. **Tool Result Accessibility**: The orchestrator's `messages` list contains the full tool results (including the OAuth URL), but the current code only uses this list as a fallback when `response_text` is empty — not to verify URL preservation.

## Correctness Properties

Property 1: Bug Condition - OAuth URL Present in Final Response

_For any_ calendar connection request where the `calendar_agent` tool successfully generates an OAuth URL (isBugCondition returns true on the unfixed code), the fixed `process_message` function SHALL return a response that contains the complete OAuth URL verbatim, so the trainer can click it to authorize calendar access.

**Validates: Requirements 2.1, 2.2, 2.3**

Property 2: Preservation - Non-Calendar Message Behavior

_For any_ input that does NOT trigger a `calendar_agent` tool call returning an OAuth URL (isBugCondition returns false), the fixed `process_message` function SHALL produce the same response as the original function, preserving all existing routing, error handling, and response generation behavior.

**Validates: Requirements 3.1, 3.2, 3.3, 3.4**

## Fix Implementation

### Changes Required

Assuming our root cause analysis is correct:

**File**: `src/services/strands_agent_service.py`

**Function**: `process_message` (post-processing of `response_text`)

**Specific Changes**:

1. **Add OAuth URL extraction helper**: Create a helper method `_extract_oauth_url_from_messages` that scans the orchestrator's `messages` list for tool results from `calendar_agent` containing OAuth URLs (matching `https://accounts.google.com/o/oauth2/` or `https://login.microsoftonline.com/`).

2. **Add post-processing URL verification**: After extracting `response_text` from the `AgentResult`, check if an OAuth URL exists in the tool results. If it does and is NOT present in `response_text`, replace `response_text` with the original `calendar_agent` tool result string (which already contains the URL in a well-formatted message).

3. **Preserve existing fallback logic**: The existing fallback that checks tool results when `response_text` is empty should remain unchanged. The new logic adds a second check specifically for OAuth URL preservation.

**File**: `src/services/strands_agent_service.py`

**Function**: `_build_domain_agent_tools` (calendar_agent tool)

**Specific Changes**:

4. **No changes to calendar_agent**: The `calendar_agent` tool function already correctly generates the OAuth URL and formats a response string. No changes needed here.

5. **No changes to orchestrator system prompt**: While the system prompt already instructs the LLM to copy the calendar_agent response verbatim, this is a soft constraint. The structural post-processing fix makes the prompt instruction a defense-in-depth measure rather than the sole mechanism.

## Testing Strategy

### Validation Approach

The testing strategy follows a two-phase approach: first, surface counterexamples that demonstrate the bug on unfixed code, then verify the fix works correctly and preserves existing behavior.

### Exploratory Bug Condition Checking

**Goal**: Surface counterexamples that demonstrate the bug BEFORE implementing the fix. Confirm or refute the root cause analysis. If we refute, we will need to re-hypothesize.

**Test Plan**: Write tests that mock the Strands Agent orchestrator to simulate the scenario where `calendar_agent` returns a response with an OAuth URL but the orchestrator's final `response_text` omits it. Run these tests on the UNFIXED code to observe failures and understand the root cause.

**Test Cases**:
1. **Google Calendar URL Lost**: Simulate orchestrator receiving `calendar_agent` result with Google OAuth URL, but generating response_text without the URL (will fail on unfixed code)
2. **Outlook Calendar URL Lost**: Simulate orchestrator receiving `calendar_agent` result with Outlook OAuth URL, but generating response_text without the URL (will fail on unfixed code)
3. **URL Partially Included**: Simulate orchestrator including a truncated or malformed URL in response_text (will fail on unfixed code)

**Expected Counterexamples**:
- `response_text` contains phrases like "clique no link acima" but no URL matching `https://accounts.google.com/` or `https://login.microsoftonline.com/`
- Possible causes: LLM paraphrasing, LLM summarizing long URLs, LLM treating the URL as context rather than content to reproduce

### Fix Checking

**Goal**: Verify that for all inputs where the bug condition holds, the fixed function produces the expected behavior.

**Pseudocode:**
```
FOR ALL input WHERE isBugCondition(input, tool_results, final_response_unfixed) DO
  result := process_message_fixed(input)
  oauth_url := extractOAuthURL(tool_results)
  ASSERT oauth_url IN result['response']
END FOR
```

### Preservation Checking

**Goal**: Verify that for all inputs where the bug condition does NOT hold, the fixed function produces the same result as the original function.

**Pseudocode:**
```
FOR ALL input WHERE NOT isBugCondition(input, tool_results, final_response) DO
  ASSERT process_message_original(input) = process_message_fixed(input)
END FOR
```

**Testing Approach**: Property-based testing is recommended for preservation checking because:
- It generates many test cases automatically across the input domain (various message types, error conditions)
- It catches edge cases that manual unit tests might miss (e.g., messages that mention "calendar" but aren't connection requests)
- It provides strong guarantees that behavior is unchanged for all non-buggy inputs

**Test Plan**: Observe behavior on UNFIXED code first for non-calendar messages, then write property-based tests capturing that behavior.

**Test Cases**:
1. **Student Message Preservation**: Verify that student management messages (register, view, update) continue to route correctly and return the same responses after the fix
2. **Session Message Preservation**: Verify that session scheduling messages continue to work correctly after the fix
3. **Payment Message Preservation**: Verify that payment messages continue to work correctly after the fix
4. **Calendar Error Preservation**: Verify that calendar connection failures (invalid provider, missing credentials) continue to return error messages without URLs
5. **Greeting Preservation**: Verify that general greetings continue to be answered directly by the orchestrator

### Unit Tests

- Test `_extract_oauth_url_from_messages` with various message structures (Google URL, Outlook URL, no URL, malformed messages)
- Test post-processing logic: response_text missing URL → URL injected from tool result
- Test post-processing logic: response_text already contains URL → no modification
- Test post-processing logic: no OAuth URL in tool results → response_text unchanged
- Test calendar connection error flow (no URL generated) → error message unchanged

### Property-Based Tests

- Generate random calendar connection requests with random OAuth state tokens and verify the URL is always present in the final response (Hypothesis strategy: random trainer_id, random provider from {google, outlook}, random state token)
- Generate random non-calendar messages and verify the post-processing does not modify the response (Hypothesis strategy: random messages from student/session/payment domains)
- Generate random OAuth URLs with varying query parameter lengths and verify they survive the post-processing intact

### Integration Tests

- Test full `process_message` flow with mocked Bedrock model for Google Calendar connection request
- Test full `process_message` flow with mocked Bedrock model for Outlook Calendar connection request
- Test full `process_message` flow with mocked Bedrock model for non-calendar messages to verify no regression
